# MVP 功能清單

> 規劃用於第一版可分發的 Windows 應用程式。原則:**能讓一個使用者從影片到檢舉輔助包全流程跑通**,其他都先砍。

## 狀態標記

- `[ ]` 未開始
- `[~]` 進行中
- `[x]` 完成

---

## P0 — MVP 必備 (缺一項都不算 MVP 完成)

### 核心流程

- [ ] **影片匯入**:拖放或檔案選擇;支援 mp4 / mov / mkv;PyAV 解碼
- [ ] **專案檔 (.zgproj)**:新增、儲存、載入、續跑
- [ ] **ROI 編輯器**:於第一幀(或使用者指定的幀)上繪製斑馬線多邊形,可新增/刪除/拖曳頂點
- [ ] **Homography 校正**:使用者標註 ≥ 4 組地面參考點並輸入實際距離(公尺),計算變換矩陣
- [ ] **Stage 1 粗篩管線**:1 fps,`yolo11n`,偵測 person + 車輛大類,找出 ROI 附近同時有人車的時段
- [ ] **Stage 2 精細管線**:對粗篩時段 5–10 fps 分析,加 ByteTrack 追蹤
- [ ] **違規規則引擎**:行人在 ROI + 車輛在 yield 距離內 + 車輛未減速 → 輸出違規候選
- [ ] **進度 UI**:可取消、背景執行、誠實 ETA、checkpoint 續跑
- [ ] **違規清單 + Timeline**:把所有違規候選標記在 timeline 上,點擊跳到該時段
- [ ] **Review Deck**:逐一顯示違規候選,影片循環播放 ±3 秒,支援採用 / 拒絕 / 調整起訖時間,鍵盤快捷鍵 A/R/E
- [ ] **片段匯出**:ffmpeg 無損剪輯違規前後 N 秒(預設 5 秒),輸出 .mp4
- [ ] **關鍵幀匯出**:違規發生瞬間的 .jpg,提供「含 bbox 標註」與「原圖」各一份
- [ ] **檢舉書草稿**:.txt 或 .pdf,自動填入時間、持續秒數、違規描述,留空位給使用者補地點、車牌

### 系統 / 部署

- [ ] **GPU 偵測與 fallback**:CUDA → DirectML → CPU,有警告視窗告知預估時間
- [ ] **模型管理**:安裝檔綁附 yolo11n,首次啟動詢問下載 yolo11m
- [ ] **綁附 ffmpeg**:安裝時自動放到 `resources/ffmpeg/`
- [ ] **首次啟動免責聲明**:使用者必須閱讀並同意方可繼續使用(見 `disclaimer.md`)
- [ ] **PyInstaller 打包**:產出單一資料夾或 single-file exe
- [ ] **Inno Setup 安裝檔**:安裝到 `%LOCALAPPDATA%\Programs\ZebraGuard\`,不需管理員權限
- [ ] **解除安裝**:乾淨移除,但詢問是否保留使用者專案檔

## P1 — 強烈建議但非死線 (MVP 可不做,但越早做越省事)

- [ ] **地點 preset**:ROI + homography 存成 preset,下次同路口影片一鍵套用
- [ ] **多違規導航快捷鍵**:J/K 上一個下一個違規事件
- [ ] **Timeline 縮放 / 雙倍速預覽**
- [ ] **日誌收集**:使用者點一個按鈕打包最近 log(協助回報 bug)
- [ ] **基本 crash dialog**:當機時顯示訊息與 log 位置而非靜默死掉
- [ ] **Benchmark 自檢**:首次啟動或設定頁可量測目前硬體的 fps,據此推估 ETA

---

## Phase 2 — 有人用了再做

- [ ] **車牌 OCR / ALPR**:自動填入檢舉書車牌欄位
- [ ] **瀏覽器輔助填單**:開啟警政署檢舉頁並把內容複製到剪貼簿
- [ ] **程式碼簽章**:Azure Trusted Signing 或 EV cert;見 `packaging.md`
- [ ] **自動更新**:check-for-update 連到 GitHub Release;**不做全自動靜默更新**
- [ ] **Crash reporting**:Sentry 或自架;嚴格 opt-in
- [ ] **i18n**:英文介面
- [ ] **多影片批次**:同一路口多段影片排隊處理
- [ ] **違規樣式自訂**:讓使用者調整 yield 距離、停止速度閾值等參數
- [ ] **夜間 / 逆光強化**:可能需要換 detection 模型或加預處理

---

## Phase 3 — 視情況而定,可能不做

- [ ] **全自動送件**:**本專案原則上不做**,法律與平台 ToS 風險過高。若做,必須使用者每次親自確認。
- [ ] **雲端同步 / 多裝置**:與「本機應用程式」定位相違,不建議。
- [ ] **行動版 companion app**:現場拍攝即傳 → 本機自動分析。需要重新評估後端。
- [ ] **車種 / 車型細分類**:需自訓模型。

## 相依關係速查

```
影片匯入 ─┬─> ROI 編輯 ──┐
         └─> Homography ┤
                        ├─> Stage 1 ─> Stage 2 ─> 違規引擎 ─> 清單/Timeline ─> Review ─> 匯出
        專案檔 (.zgproj) 貫穿以上所有步驟,每一步都要能存取與更新
```

**可以平行動手**:ROI 編輯器、Homography 校正、ml/ 核心演算法三者之間相依度低,可同時開發。UI 骨架(MainWindow + 空殼 dialog)最好最早做,才能早點把各模組串起來。

## 完成 MVP 的判斷標準

以下情境能全程跑通,就算 MVP 完成:

1. 我開啟 app,點「新專案」,選一段 1 小時的路口影片
2. 我在第一幀畫了斑馬線,標了 4 個地面點並輸入距離
3. 我按「開始分析」,看到 ETA,關掉 app,重開,分析繼續
4. 分析完,我看到 timeline 上有 5 筆違規候選
5. 我逐一審查,採用 3 筆、拒絕 2 筆
6. 我按「匯出」,得到一個資料夾,裡面有 3 個 .mp4 片段 + 3 組關鍵幀 + 3 份檢舉書草稿
7. 我可以拿著這些東西去警政署網頁人工提報

只要這個故事走得通,MVP 就上線。其他都是 Phase 2+。
