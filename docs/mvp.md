# MVP 功能清單

> 規劃用於第一版可分發的 Windows 應用程式。原則:**能讓一個使用者從行車記錄器影片到檢舉輔助包全流程跑通**,其他都先砍。

## 狀態標記

- `[ ]` 未開始
- `[~]` 進行中
- `[x]` 完成

---

## 範圍鎖定

- **僅支援行車記錄器影片**(dashcam)。不做固定式監視器 / 路口 CCTV。
- **不提供手動 ROI 編輯器、不做 Homography 校正**。斑馬線位置由 Mask2Former 於背景以高頻(每數幀一次)自動偵測;距離判斷以**像素**為單位,不換算成公尺。
- 違規判定邏輯見 `legal-rules.md` 與 `scripts/presets/v7_baseline_*`(MVP 使用 v7 預設參數)。

## P0 — MVP 必備 (缺一項都不算 MVP 完成)

### 核心流程

- [ ] **影片匯入**:拖放或檔案選擇;支援 mp4 / mov / mkv
- [ ] **專案檔 (.zgproj)**:新增、儲存、載入、續跑
- [ ] **單階段分析管線 (v7 預設)**:Mask2Former 抓斑馬線 mask + YOLO11 + ByteTrack + 違規規則引擎一氣呵成;使用 `scripts/presets/v7_baseline_params.json` 的參數
- [ ] **Ego-vehicle 過濾**:dashcam 自車(車頭/引擎蓋反射)自動排除
- [ ] **騎士 vs 行人區分**:騎在機車/自行車上的人不計為穿越行人
- [ ] **進度 UI**:可取消、背景執行、誠實 ETA、checkpoint 續跑
- [ ] **違規清單 + Timeline**:把所有違規候選標記在 timeline 上,點擊跳到該時段
- [ ] **Review Deck**:逐一顯示違規候選,影片循環播放 ±3 秒,支援採用 / 拒絕 / 調整起訖時間,鍵盤快捷鍵 A/R/E
- [ ] **片段匯出**:ffmpeg 無損剪輯違規前後 N 秒(預設 5 秒),輸出 .mp4
- [ ] **關鍵幀匯出**:違規發生瞬間的 .jpg,提供「含 bbox + mask 標註」與「原圖」各一份
- [ ] **檢舉書草稿**:.txt 或 .pdf,自動填入時間、持續秒數、違規描述,留空位給使用者補地點、車牌

### 系統 / 部署

- [ ] **GPU 偵測與 fallback**:CUDA → DirectML → CPU,有警告視窗告知預估時間(Mask2Former 在 CPU 上極慢,須明確警示)
- [ ] **模型管理**:安裝檔綁附 yolo11n;Mask2Former 權重(~1 GB)於首次啟動下載,可自選 CDN
- [ ] **綁附 ffmpeg**:安裝時自動放到 `resources/ffmpeg/`
- [ ] **首次啟動免責聲明**:使用者必須閱讀並同意方可繼續使用(見 `disclaimer.md`)
- [ ] **PyInstaller 打包**:產出單一資料夾或 single-file exe
- [ ] **Inno Setup 安裝檔**:安裝到 `%LOCALAPPDATA%\Programs\ZebraGuard\`,不需管理員權限
- [ ] **解除安裝**:乾淨移除,但詢問是否保留使用者專案檔

## Phase 2 — 有人用了再做

- [ ] **車牌 OCR / ALPR**:自動填入檢舉書車牌欄位
- [ ] **瀏覽器輔助填單**:開啟警政署檢舉頁並把內容複製到剪貼簿
- [ ] **程式碼簽章**:Azure Trusted Signing 或 EV cert;見 `packaging.md`
- [ ] **自動更新**:check-for-update 連到 GitHub Release;**不做全自動靜默更新**
- [ ] **Crash reporting**:Sentry 或自架;嚴格 opt-in
- [ ] **i18n**:英文介面
- [ ] **多影片批次**:多段影片排隊處理
- [ ] **違規樣式自訂**:進階使用者可調整 dilate_px、moving_px、person_conf 等 v7 參數
- [ ] **夜間 / 逆光強化**:可能需要換 detection 模型或加預處理
- [ ] **固定式 CCTV 支援**:若未來要做,需重新加回 ROI 編輯器與可選的 homography 校正(**已明確排除於本專案 MVP 範圍外**)

---

## Phase 3 — 視情況而定,可能不做

- [ ] **全自動送件**:**本專案原則上不做**,法律與平台 ToS 風險過高。若做,必須使用者每次親自確認。
- [ ] **雲端同步 / 多裝置**:與「本機應用程式」定位相違,不建議。
- [ ] **行動版 companion app**:現場拍攝即傳 → 本機自動分析。需要重新評估後端。
- [ ] **車種 / 車型細分類**:需自訓模型。

---

## 相依關係速查

```
影片匯入 ──> 單階段分析 (Mask2Former + YOLO + 規則) ──> 清單/Timeline ──> Review ──> 匯出
           專案檔 (.zgproj) 貫穿以上所有步驟,每一步都要能存取與更新
```

**可以平行動手**:UI 骨架、背景 worker、匯出層三者相依度低,可同時開發。分析管線本身已由 `scripts/zero_shot_detect.py` 的 v7 preset 完成,UI 只需在 QThread 中呼叫並串接進度。

## 完成 MVP 的判斷標準

以下情境能全程跑通,就算 MVP 完成:

1. 我開啟 app,點「新專案」,選一段 10 分鐘的行車記錄器影片
2. 我按「開始分析」,看到 ETA,關掉 app,重開,分析繼續
3. 分析完,我看到 timeline 上有 N 筆違規候選
4. 我逐一審查,採用 3 筆、拒絕其他
5. 我按「匯出」,得到一個資料夾,裡面有 3 個 .mp4 片段 + 3 組關鍵幀 + 3 份檢舉書草稿
6. 我可以拿著這些東西去警政署網頁人工提報

只要這個故事走得通,MVP 就上線。其他都是 Phase 2+。
